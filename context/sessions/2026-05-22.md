# 2026-05-22

## Logs Provider Work

- Added core `logs watch` and `logs wait` loops on top of the bounded
  `LogProvider::search` contract.
- `watch` emits Conduit-owned stream events as compact text or JSONL, dedupes
  repeated poll results, and renders logs oldest-first within each poll.
- `wait` exits `0` when a matching log appears and returns structured
  `status: timeout` output with exit code `1` when the timeout expires.
- Watch and wait default to `--since now`; bounded filters such as `--date` and
  `--to` keep their fixed range instead of advancing to the local clock.
- Fixture logs honor the resolved time range, so fixture examples should use
  `--date 2026-05-22` when they need deterministic matching events.
- Extended `plugin check` to validate logs provider components by path or from
  project config, giving logs plugins the same metadata validation and
  cache-warming path as OpenAPI plugins.
- Added `secret-store-v1` for logs plugins and exact `secrets.names`
  capability grants. Secrets are stored in a user-scoped Conduit state location
  with restrictive permissions on Unix.
- Added the provider-neutral `logs auth` command and `authenticate` logs
  provider entry point. `--secret-stdin` lets a plugin accept externally
  acquired auth material without Conduit rendering it.
- Added `conduit logs auth --check`. The core passes an explicit auth-check
  flag to logs providers, rejects combining it with `--secret-stdin`, and
  exits non-zero when the provider reports `action_required`.
- Added repeatable negative log filters: `--exclude-message`,
  `--exclude-logger`, and the Java-friendly alias `--exclude-class`.
- Backend-specific logs adapters should live outside Conduit core and map the
  normalized logs contract to their own dashboard/query/auth systems.
